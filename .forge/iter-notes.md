## Decisions made this iter
- **Claim-first ordering for QuestionTimeoutService**: the new order is `MarkTimedOutAsync` (atomic CAS) → publish → edit → audit. The old order was `GetExpiredAsync` → publish → edit → audit → mark. Evaluator item 5 was right that the old order let two sweepers double-publish a single row. The trade is at-most-once-from-sweeper: if a crash happens between claim and publish, the row is `TimedOut` with no decision event. Documented in the `<remarks>` block as recoverable via an audit-gap sweep (TimedOut rows lacking a matching HumanResponseAuditEntry). Operationally this is fine because the operator still gets a "default action applied" Telegram message even if the agent doesn't see the event — they can re-trigger.
- **EF Core 8 `ExecuteUpdateAsync` with `HasConversion<string>()` enum mapping**: works without ceremony — EF translates `Where(x => x.Status == PendingQuestionStatus.Pending || x.Status == PendingQuestionStatus.AwaitingComment)` into `WHERE Status IN ('Pending', 'AwaitingComment')` (provider-neutral). Confirmed by `dotnet build` + the persistent-store test suite passing. No raw SQL.
- **Sender throws instead of swallowing**: the previous log+continue pattern silently dropped the load-bearing pending row. New `PendingQuestionPersistenceException` carries enough state for Stage 4.1's `OutboundQueueProcessor` to (a) NOT re-send the Telegram message (it's already delivered) and (b) retry just the StoreAsync (the upsert is idempotent on `QuestionId`). The `<remarks>` on the new exception type spell this out.
- **DefaultActionId index needs all four files in sync**: configuration, migration `Up`, designer snapshot, and the live ModelSnapshot. Forgetting any one of them leaves a `dotnet ef migrations add` foot-gun (the next migration would re-generate the index, producing a duplicate `CreateIndex` call). All four updated this iter.
- **Drop the open-question hard gate (evaluator item 1)**: prior iter surfaced an open question about TelegramMessageSender vs Stage 4.1 OutboundQueueProcessor ownership of `PersistAsync`. This iter commits to the brief's wording — the sender persists directly, and a one-line comment at the call site explicitly notes that Stage 4.1 may revisit this. No `openQuestions` JSON block in the reply.
